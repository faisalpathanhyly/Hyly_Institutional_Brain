# Client Proof Story Schema

## Purpose

Client Proof Stories are S.Academy learning assets that show how Hyly solved real client problems.

They are not website testimonials, client reviews, or generic case studies. They are structured problem-solving stories that help Hyly teams answer:

```text
Have we solved a problem like this before?
```

The story should lead with the client problem, not client praise.

---

## Standard Location

```text
s_academy/client_intelligence/client_proof_stories/stories/
```

Each story should be saved as one Markdown file.

### File Naming Format

```text
client_name_problem_area.md
```

---

## When To Create A Client Proof Story

Create a Client Proof Story when Hyly has enough information to explain:

```text
Client context
Problem
Pain points
Bottlenecks
What Hyly identified
Product solution
How the solution worked
Why the solution worked
How the obstacle was tackled
Outcome or impact
Reusable pattern for future clients
```

If the outcome is not yet known, use a Pre-Pilot Client Knowledge Asset instead.

---

## Writing Direction

A Client Proof Story should be practical, specific, and reusable.

It should not sound like:

```text
The client loved Hyly.
```

It should sound like:

```text
The client had this problem. Hyly identified this bottleneck. This product capability solved it this way. Future clients with the same issue can learn from this pattern.
```

---

## Outcome Rules

Use documented outcomes only.

If there are no confirmed numbers, write the outcome qualitatively.

### Acceptable qualitative outcome

```text
The client gained a clearer way to review reporting performance across systems.
```

### Avoid unsupported claims

```text
The client transformed operations and improved results dramatically.
```

Do not invent metrics, timelines, or business impact.

---

## Media Rules

Media should be uploaded to Cloudinary or Google Drive.

Git should contain only the media link.

The media can be a video, carousel, screen recording, visual walkthrough, deck, or notes file. The story file should still contain the structured breakdown, even when media is attached.

---

## Completion Checklist

Before saving a Client Proof Story, confirm:

```text
[ ] The story is problem-led, not praise-led
[ ] Client context is clear
[ ] Problem is specific
[ ] Pain points are listed
[ ] Bottlenecks are identified
[ ] Hyly diagnosis is explained
[ ] Product solution is connected to the problem
[ ] Outcome or impact is documented honestly
[ ] Reusable pattern is clear
[ ] Media link is added if available
[ ] Internal notes are included if needed
[ ] Knowledge check is included
```
